= Transaction in SQL and Spring

Transaction is group of related sql statements that works as single unit and inherits ADIC property. It is used to maintain data accuracy and integrity.


== *_What are ACID Properties?_*

ACID properties are a set of rules that ensure reliable and safe processing of database transactions.  
ACID stands for:

* A → Atomicity  
* C → Consistency  
* I → Isolation  
* D → Durability  

These properties help maintain data accuracy and integrity.

---

=== 1. Atomicity

Atomicity means that a transaction is treated as a *single unit*.  
Either **all operations succeed**, or **none of them happen**. 
If any step in a transaction fails, the entire transaction is rolled back.

==== Example
Money transfer from Account A to Account B:

* Step 1: Deduct ₹500 from Account A  
* Step 2: Add ₹500 to Account B  

If Step 2 fails, Step 1 is undone. So, no money is lost.

[source,java]
----
BEGIN TRANSACTION;
UPDATE account SET balance = balance - 500 WHERE id = 1;
UPDATE account SET balance = balance + 500 WHERE id = 2;
COMMIT;
----

---

=== 2. Consistency

Consistency ensures that the database moves from **one valid state to another valid state**. After a transaction, all database rules, constraints, and validations must be satisfied.

==== Example
If account balance cannot be negative:

* Before transfer: Balance = ₹1000  
* Transfer amount = ₹1500 → Not allowed  

The transaction fails, and the database remains unchanged.

[source,sql]
----
CHECK (balance >= 0)
----

This ensures invalid data is never saved.

---

=== 3. Isolation

Isolation ensures that **multiple transactions can run at the same time without affecting each other**. Each transaction works as it is the only one running.

==== Example
Two users trying to book the *last train seat* at the same time:

* Only **one booking succeeds**
* The other transaction fails or waits

This prevents *dirty reads* and *data corruption*.

---

=== 4. Durability

Durability guarantees that **once a transaction is committed, it stays saved permanently**, even after system crashes. 
Saved data will not be lost due to power failure or server restart.

==== Example
After confirming a payment:

* Data is written to disk  
* Even if the server crashes, the payment record remains safe  

This is achieved using *logs, backups, and disk storage*.

---

=== Isolation Levels

Isolation levels define how strictly transactions are separated from each other. They prevent three common read problems:

1. **Dirty Read**: Reading uncommitted data from another transaction.
2. **Non-Repeatable Read**: Reading the same row twice but getting different results because another transaction updated it.
3. **Phantom Read**: Reading a set of rows twice but getting different results because another transaction inserted/deleted rows.

Here are the 4 standard isolation levels from least strict to most strict:

==== 1. READ_UNCOMMITTED
* **Meaning**: A transaction can read data that is not yet committed by other transactions.
* **Problem**: Can lead to **Dirty Reads**.
* **Example**: Transaction A updates an account balance from ₹1000 to ₹500 but hasn't committed. Transaction B reads the balance as ₹500. Transaction A rolls back. Transaction B now has wrong data.

==== 2. READ_COMMITTED (Default in PostgreSQL)
* **Meaning**: A transaction can only read data that has been committed.
* **Problem**: Prevents Dirty Reads, but can lead to **Non-Repeatable Reads**.
* **Example**: Transaction A reads account balance as ₹1000. Transaction B updates it to ₹500 and commits. Transaction A reads it again and gets ₹500. The read changed during the same transaction.

==== 3. REPEATABLE_READ (Default in MySQL InnoDB)
* **Meaning**: If a transaction reads a row, it will see the exact same data if it reads it again, even if another transaction updates and commits it.
* **Problem**: Prevents Dirty and Non-Repeatable reads, but can lead to **Phantom Reads**.
* **Example**: Transaction A reads all "active" users and finds 5. Transaction B inserts a new active user and commits. If Transaction A reads again, it still sees 5. But if it tries to update all active users, 6 rows might be updated (the 6th is the "phantom").

==== 4. SERIALIZABLE
* **Meaning**: The strictest level. Transactions run as if they are executed one after the other (sequentially).
* **Problem**: Prevents all read problems, but is the slowest and can cause performance bottlenecks or deadlocks.
* **Example**: Two users trying to book tickets at the exact same time. The database forces User B to wait completely until User A's transaction finishes.

[cols="2,1,1,1", options="header"]
|===
| Isolation Level | Dirty Read | Non-Repeatable Read | Phantom Read

| READ_UNCOMMITTED | Yes | Yes | Yes
| READ_COMMITTED   | No  | Yes | Yes
| REPEATABLE_READ  | No  | No  | Yes
| SERIALIZABLE     | No  | No  | No
|===

== *_Spring Boot Transactions & Propagation Types_*

*Transaction* is group of related sql statements that works as single unit and inherits ADIC property. It is used to maintain data accuracy and integrity. It make sure that eather:

* all succeed, or
* all fail together

Spring mainly uses the `@Transactional` annotation to manage this.

[source,java]
----
@Transactional
public void transferMoney() {
    debitAccount();
    creditAccount();
}
----

If any exception occurs, Spring automatically performs a *rollback*.

---

== @Transactional Annotation Basics

[source,java]
----
@Transactional(
    propagation = Propagation.REQUIRED,
    isolation = Isolation.READ_COMMITTED,
    rollbackFor = Exception.class
)
----

Common attributes:

* `propagation` → How transactions interact with each other
* `isolation` → Data visibility between transactions
* `rollbackFor` → Which exceptions trigger rollback

---

== *_Transaction Propagation_*

Propagation defines **how a method behaves when it is called inside another transaction**.

Think of it as:
> “Should I join the existing transaction, create a new one, or run without one?”

[source,java]
----
@Transactional
public void placeOrder() {
    savePayment();
}

@Transactional
// This method is called inside another transaction so how does it behave?
public void saveOrder() {
    orderRepository.save(...);
}

public void savePayment() {
    paymentRepository.save(...);
}
----


---

== 1. REQUIRED (Default)

=== Meaning
* Uses existing transaction if available
* Otherwise, creates a new transaction

=== Example
[source,java]
----
@Transactional(propagation = Propagation.REQUIRED)
public void placeOrder() {
    saveOrder();
    savePayment();
}
----

=== Flow
* If `placeOrder()` is called inside another transaction → joins it
* If not → starts a new transaction

=== Real-life Example
Placing an order and saving payment must succeed or fail together.

---

== 2. REQUIRES_NEW

=== Meaning
* Always creates a **new transaction**
* Suspends the existing transaction

=== Example
[source,java]
----
@Transactional(propagation = Propagation.REQUIRES_NEW)
public void saveAuditLog() {
    auditRepository.save(...);
}
----

=== Flow
* Parent transaction is paused
* New transaction starts
* Commits independently

=== Real-life Example
Audit logs should be saved even if the main transaction fails.

---

== 3. SUPPORTS

=== Meaning
* Joins existing transaction if present
* Otherwise, runs *without* a transaction

=== Example
[source,java]
----
@Transactional(propagation = Propagation.SUPPORTS)
public List<User> getUsers() {
    return userRepository.findAll();
}
----

=== Use Case
Read-only operations that don't require strict transactional behavior.

---

== 4. NOT_SUPPORTED

=== Meaning
* Always runs **without a transaction**
* Suspends any existing transaction

=== Example
[source,java]
----
@Transactional(propagation = Propagation.NOT_SUPPORTED)
public void generateReport() {
    // long-running read-only logic
}
----

=== Use Case
Reporting, analytics, or non-DB operations.

---

== 5. MANDATORY

=== Meaning
* Must run inside an existing transaction
* Throws exception if none exists

=== Example
[source,java]
----
@Transactional(propagation = Propagation.MANDATORY)
public void updateInventory() {
    inventoryRepository.save(...);
}
----

=== Use Case
Critical methods that should never run outside a transaction.

---

== 6. NEVER

=== Meaning
* Must NOT run inside a transaction
* Throws exception if a transaction exists

=== Example
[source,java]
----
@Transactional(propagation = Propagation.NEVER)
public void sendNotification() {
    // external service call
}
----

=== Use Case
External API calls where transactions are not allowed.

---

== 7. NESTED

=== Meaning
* Runs inside a parent transaction
* Can rollback independently using savepoints

=== Example
[source,java]
----
@Transactional(propagation = Propagation.NESTED)
public void saveChildRecord() {
    childRepository.save(...);
}
----

=== Use Case
Partial rollback without affecting the parent transaction.

NOTE: Works only with databases that support *savepoints* (e.g., MySQL InnoDB).

---

== How @Transactional Works Internally (The Magic Explained)

Spring's `@Transactional` seems like magic, but it's powered by a design pattern called **AOP (Aspect-Oriented Programming)** and **proxies**. Understanding this is key to avoiding common pitfalls.

=== The Core Idea: AOP and Proxies

1.  **Scanning**: When your Spring application starts, it scans for beans with `@Transactional` methods.
2.  **Proxy Creation**: Instead of giving you the actual bean object, Spring creates a **proxy** object that wraps your bean. This proxy has the same methods as your bean.
3.  **Interception**: When you call a method on that bean, you're actually calling the proxy. The proxy intercepts this call.

This proxy is where the transaction magic happens. It's like having a security guard (the proxy) who checks your clearance (transaction rules) before letting you into a room (the actual method).

[source,text]
----
Your Code  ───calls───►  Proxy Bean  ───manages transaction───►  Original Bean
                                       (starts, commits, or          (your actual code)
                                        rolls back)
----

=== Step-by-Step Workflow

Here’s what happens when you call a `@Transactional` method:

1.  **Method Call Intercepted**: The proxy intercepts the call to your method (e.g., `placeOrder()`).
2.  **Transaction Logic Begins**: The proxy's transactional "advice" (code) runs *before* your method.
3.  **Check Propagation**: It checks the `propagation` setting. Does an existing transaction exist? Should it create a new one? (e.g., `REQUIRED` will join an existing one or create a new one).
4.  **Start Transaction**: The proxy asks the `TransactionManager` to begin a transaction. This typically involves getting a database connection and setting `auto-commit` to `false`.
5.  **Execute Your Method**: The proxy calls the *original* method on your actual bean. Your business logic runs, performing database operations.
6.  **Handle the Outcome**:
    *   **Success**: If your method finishes without throwing an exception, the proxy's transactional advice runs again. It tells the `TransactionManager` to `commit()` the transaction.
    *   **Failure**: If your method throws a `RuntimeException` (or an exception specified in `rollbackFor`), the proxy catches it. It tells the `TransactionManager` to `rollback()` the transaction, undoing all changes. The exception is then re-thrown.
7.  **Return Control**: The proxy returns the result (or exception) to the original caller.

=== Why @Transactional Can Fail: Common Pitfalls

Because `@Transactional` relies on proxies, it has some limitations that can be confusing.

==== 1. Self-Invocation (Internal Method Calls)

This is the most common pitfall. If you call a `@Transactional` method from another method *within the same class*, the transaction will **NOT** be applied.

[source,java]
----
@Service
public class MyService {

    public void externalMethod() {
        // This call bypasses the proxy!
        this.internalTransactionalMethod(); // BAD
    }

    @Transactional
    public void internalTransactionalMethod() {
        // This code will NOT run in a transaction when called from externalMethod()
    }
}
----

*Why it fails*: When `externalMethod()` calls `internalTransactionalMethod()`, it uses the `this` reference, which points to the original object, not the proxy. The proxy is completely bypassed, so no transaction is started.

*Solution*: Call the method from a different bean, or inject the service into itself to call the method on the proxy.

==== 2. Non-Public Methods

`@Transactional` only works on `public` methods by default. Spring will silently ignore the annotation on `private`, `protected`, or package-private methods.

*Why it fails*: The proxy-based AOP is configured to only intercept public method calls for security and design reasons.

==== 3. Exception Rollback Rules

By default, Spring only rolls back transactions for **unchecked exceptions** (`RuntimeException` and `Error`). It does **NOT** roll back for **checked exceptions** (like `IOException`, `SQLException`).

*Why*: This is an old design choice from the EJB era, where checked exceptions were seen as recoverable business exceptions, not system failures.

*Solution*: Use the `rollbackFor` attribute to tell Spring to roll back on checked exceptions.

== Important Notes (Very Common Interview Points)

* `@Transactional` works only on **public methods**
* Internal method calls (same class) do NOT trigger transactions
* Runtime exceptions cause rollback by default
* Checked exceptions need `rollbackFor`

[source,java]
----
@Transactional(rollbackFor = Exception.class)
----
