== Introduction

*Java 9* (September 2017) introduced the *Java Platform Module System* (Project Jigsaw), one of the most significant changes to the Java platform since its inception. Beyond modules, Java 9 brought practical enhancements to existing APIs and several developer productivity improvements.

*Key Themes:*

- *Modularity*: Java Platform Module System (JPMS) for better encapsulation
- *API Enhancements*: Collection factories, Stream improvements, Optional enhancements
- *Developer Productivity*: JShell REPL, improved try-with-resources
- *Interface Evolution*: Private methods in interfaces

*Interview Importance:* ⭐⭐⭐⭐

Java 9's *Collection factory methods* and *Stream/Optional improvements* are frequently used in modern codebases and appear in interviews. The Module System is important for architectural discussions, though less common in day-to-day coding interviews.

- link:examples/java9/Java9CollectionsDemo.java[Java9CollectionsDemo.java] - Collection factories, Stream, and Optional improvements
- link:examples/java9/Java9InterfaceDemo.java[Java9InterfaceDemo.java] - Private interface methods with real-world scenarios

== Module System (Project Jigsaw)

==== Java Platform Module System (JPMS)

.Overview / What it is
[%collapsible]
====
The *Module System* introduces a higher level of aggregation above packages. A *module* is a self-describing collection of code and data with explicit dependencies on other modules. Think of it as *organizing your application into large-scale components with clear boundaries*.

*Core Concept:*

Before Java 9, the JDK was a monolithic platform. With modules, you can create applications that only include the parts of the JDK they actually use, and you can enforce strong encapsulation at the module level.

[source,java]
----
// module-info.java - Module descriptor
module com.myapp.userservice {
    // What this module depends on
    requires java.sql;
    requires com.myapp.common;

    // What this module exposes to others
    exports com.myapp.userservice.api;

    // What stays hidden (strong encapsulation)
    // com.myapp.userservice.internal is NOT exported
}
----

*Key Concepts:*

- *Module*: A named collection of packages and resources
- *module-info.java*: Module descriptor (placed at root of module)
- *requires*: Declares dependencies on other modules
- *exports*: Makes packages available to other modules
- *opens*: Allows reflection access to a package
- *provides/uses*: Service provider/consumer declarations

*Mental Model:*

Think of modules as *contracts between large components*. Just like interfaces define contracts between classes, modules define contracts between major parts of your application. They answer: "What do I need?" (requires) and "What do I provide?" (exports).

*Module Types:*

1. *Named modules*: Have a module-info.java descriptor
2. *Automatic modules*: JARs without module-info (transitional)
3. *Unnamed module*: Classpath-based code (legacy compatibility)
====

.Why use it
[%collapsible]
====
*1. Strong Encapsulation*

Hide internal implementation details at the module level:

[source,java]
----
// Without modules - all public classes are accessible
package com.myapp.internal;
public class InternalHelper { }  // Anyone can use this!

// With modules - control what's accessible
module com.myapp {
    exports com.myapp.api;  // Only api package is accessible
    // com.myapp.internal is hidden from outside
}
----

*2. Reliable Configuration*

Detect missing dependencies at startup, not at runtime:

[source,java]
----
// Old way - fails at runtime when class is loaded
// Exception in thread "main" java.lang.NoClassDefFoundError

// With modules - fails at startup
// Error: Module X requires Y which is missing
----

*3. Smaller Runtime Images*

Include only the modules your application needs:

[source,bash]
----
# Old way - entire JRE (~200MB)
java -jar myapp.jar

# With jlink - custom runtime image (~40MB)
jlink --module-path $JAVA_HOME/jmods:mods \
      --add-modules com.myapp \
      --output myapp-runtime
----

*4. Better Security*

Internal APIs (like `sun.misc.Unsafe`) are strongly encapsulated and can't be accessed accidentally.

*5. Scalable Platform*

Java platform itself is modularized (java.base, java.sql, java.xml, etc.), making it more maintainable.
====

.When not to use
[%collapsible]
====
*❌ DON'T Use for:*

*1. Simple Applications*

[source,java]
----
// Small apps - overhead not worth it
// Just use classpath for simple projects
javac -cp lib/* src/*.java
java -cp "lib/*:." Main
----

*2. Rapid Prototyping*

Modules add complexity that slows down quick experimentation.

*3. Applications Requiring Deep Reflection*

Some frameworks (older versions of Spring, Hibernate) heavily use reflection and may conflict with module encapsulation.

*4. When Dependencies Don't Support Modules*

If your dependencies aren't module-aware, you'll deal with automatic modules and lose some benefits.

*5. Legacy Codebases*

Retrofitting modules into large legacy codebases is often not worth the effort. Classpath-based approach still works fine.

*General Rule:* Use modules for *large, long-lived applications* where strong encapsulation and explicit dependencies add value. For small projects, libraries, or rapid prototyping, stick with the classpath.
====

.Best practices
[%collapsible]
====
*✅ DO:*

- *Start with automatic modules* before creating full module descriptors
[source,java]
----
// Add Automatic-Module-Name to MANIFEST.MF first
Automatic-Module-Name: com.myapp.userservice
----

- *One module per major component* (don't over-modularize)
[source,java]
----
// Good module structure:
// - com.myapp.core
// - com.myapp.api
// - com.myapp.persistence
----

- *Export only API packages*
[source,java]
----
module com.myapp {
    exports com.myapp.api;      // ✅ Public API
    exports com.myapp.spi;      // ✅ Service Provider Interface
    // com.myapp.internal hidden ✅
}
----

- *Use `requires transitive` for API dependencies*
[source,java]
----
module com.myapp.api {
    // If your API returns types from another module
    requires transitive java.sql;  // Users get java.sql automatically
}
----

- *Use `opens` sparingly for reflection*
[source,java]
----
module com.myapp {
    opens com.myapp.model to com.fasterxml.jackson.databind;  // For JSON serialization
}
----

*❌ DON'T:*

- *Don't export everything*
[source,java]
----
// BAD - defeats purpose of modules
module com.myapp {
    exports com.myapp.api;
    exports com.myapp.internal;  // ❌ Don't expose internals
    exports com.myapp.util;      // ❌ Keep utilities private
}
----

- *Don't create circular dependencies*
[source,java]
----
// BAD - modules can't have cycles
module A { requires B; }
module B { requires A; }  // ❌ Compilation error
----

- *Don't use split packages* (same package in multiple modules)
[source,java]
----
// BAD - same package in two modules
module A { exports com.myapp.common; }
module B { exports com.myapp.common; }  // ❌ Not allowed
----
====

.Example
[%collapsible]
====
*Basic Module Structure:*

[source,java]
----
// === Project Structure ===
myapp/
├── com.myapp.api/
│   └── module-info.java
│   └── com/myapp/api/
│       └── UserService.java
├── com.myapp.core/
│   └── module-info.java
│   └── com/myapp/core/
│       ├── UserServiceImpl.java
│       └── internal/
│           └── UserValidator.java
└── com.myapp.app/
    └── module-info.java
    └── com/myapp/app/
        └── Main.java

// === com.myapp.api/module-info.java ===
module com.myapp.api {
    // This is the API module - no dependencies
    exports com.myapp.api;
}

// === com.myapp.api/com/myapp/api/UserService.java ===
package com.myapp.api;

public interface UserService {
    User findById(long id);
    void save(User user);
}

// === com.myapp.core/module-info.java ===
module com.myapp.core {
    requires com.myapp.api;        // Depends on API
    requires java.sql;             // Uses JDBC

    exports com.myapp.core;        // Export implementation package
    // com.myapp.core.internal stays hidden
}

// === com.myapp.core/com/myapp/core/UserServiceImpl.java ===
package com.myapp.core;

import com.myapp.api.UserService;
import com.myapp.core.internal.UserValidator;

public class UserServiceImpl implements UserService {
    private final UserValidator validator = new UserValidator();

    @Override
    public User findById(long id) {
        validator.validateId(id);
        // Implementation
    }
}

// === com.myapp.core/com/myapp/core/internal/UserValidator.java ===
package com.myapp.core.internal;

// This class is NOT exported - only visible within com.myapp.core module
public class UserValidator {
    public void validateId(long id) {
        if (id <= 0) throw new IllegalArgumentException("Invalid ID");
    }
}

// === com.myapp.app/module-info.java ===
module com.myapp.app {
    requires com.myapp.api;
    requires com.myapp.core;
}

// === com.myapp.app/com/myapp/app/Main.java ===
package com.myapp.app;

import com.myapp.api.UserService;
import com.myapp.core.UserServiceImpl;
// import com.myapp.core.internal.UserValidator;  // ❌ Compilation error - not exported!

public class Main {
    public static void main(String[] args) {
        UserService service = new UserServiceImpl();
        User user = service.findById(123);
    }
}
----

*Compiling and Running:*

[source,bash]
----
# Compile modules
javac -d mods --module-source-path src $(find src -name "*.java")

# Run modular application
java --module-path mods -m com.myapp.app/com.myapp.app.Main

# Create custom runtime image
jlink --module-path $JAVA_HOME/jmods:mods \
      --add-modules com.myapp.app \
      --output myapp-image \
      --launcher myapp=com.myapp.app/com.myapp.app.Main

# Run with custom image
./myapp-image/bin/myapp
----

.Common pitfalls
[%collapsible]
====
*1. Split packages across modules*

[source,java]
----
// PITFALL - same package in multiple modules

// module A
module com.myapp.core {
    exports com.myapp.util;  // ❌
}

// module B
module com.myapp.extra {
    exports com.myapp.util;  // ❌ Same package name!
}

// FIX - use unique package names per module
module com.myapp.core {
    exports com.myapp.core.util;  // ✅
}
module com.myapp.extra {
    exports com.myapp.extra.util;  // ✅
}
----

*2. Circular dependencies*

[source,java]
----
// PITFALL - circular requires

module A {
    requires B;
}
module B {
    requires A;  // ❌ Circular dependency!
}

// FIX - extract common parts to a new module
module A {
    requires common;
}
module B {
    requires common;
}
module common {
    // Shared code
}
----

*3. Forgetting `requires transitive` for API dependencies*

[source,java]
----
// PITFALL - consumers need to repeat dependencies

// com.myapp.api
module com.myapp.api {
    requires java.sql;  // ❌ Not transitive
    exports com.myapp.api;
}

package com.myapp.api;
import java.sql.Connection;
public interface UserDao {
    Connection getConnection();  // Returns java.sql type
}

// Consumer must also require java.sql
module consumer {
    requires com.myapp.api;
    requires java.sql;  // ❌ Shouldn't need this
}

// FIX - use requires transitive
module com.myapp.api {
    requires transitive java.sql;  // ✅ Propagates to consumers
    exports com.myapp.api;
}

module consumer {
    requires com.myapp.api;  // ✅ Gets java.sql automatically
}
----

*4. Over-using `opens` for reflection*

[source,java]
----
// PITFALL - opening entire module

module com.myapp {
    opens com.myapp;  // ❌ Opens everything!
}

// FIX - open specific packages
module com.myapp {
    opens com.myapp.model to com.fasterxml.jackson.databind;  // ✅ Specific
}

// Or use --add-opens at runtime for legacy frameworks
java --add-opens com.myapp/com.myapp.internal=ALL-UNNAMED -m com.myapp
----

*5. Not handling automatic modules properly*

[source,java]
----
// PITFALL - depending on JAR without module-info

module com.myapp {
    requires some.legacy.jar;  // ❌ Name derived from JAR filename (unreliable)
}

// FIX - check automatic module name or add to manifest
// In some-legacy-jar's MANIFEST.MF:
Automatic-Module-Name: some.legacy.jar
----

*6. Mixing classpath and module path*

[source,bash]
----
# PITFALL - confusing classpath and module path

javac -cp libs/* src/*.java  # ❌ Mixing concerns
java -cp "libs/*:." --module-path mods -m myapp  # ❌ Confusion

# FIX - choose one approach
# Module path (modular app):
javac -p mods --module-source-path src $(find src -name "*.java")
java -p mods -m com.myapp/com.myapp.Main

# Classpath (traditional app):
javac -cp libs/* src/**/*.java
java -cp "libs/*:bin" com.myapp.Main
----

*7. Forgetting module-info.java in package root*

[source,java]
----
// PITFALL - wrong location

src/
└── com/myapp/
    └── module-info.java  // ❌ Wrong location!

// FIX - module-info.java must be at module root
src/
├── module-info.java  // ✅ Correct
└── com/myapp/
    └── Main.java
----

*8. Not declaring service providers correctly*

[source,java]
----
// PITFALL - service not found at runtime

module provider {
    exports com.myapp.spi;
    // Missing provides clause
}

// FIX - declare service provider
module provider {
    exports com.myapp.spi;
    provides com.myapp.spi.UserService
        with com.myapp.impl.UserServiceImpl;  // ✅
}

module consumer {
    requires com.myapp.spi;
    uses com.myapp.spi.UserService;  // ✅
}
----

*9. Assuming all JDK APIs are in java.base*

[source,java]
----
// PITFALL - missing required JDK modules

import java.sql.Connection;  // ❌ Compilation fails

module-info.java:
module com.myapp {
    // Missing requires java.sql
}

// FIX - explicitly require JDK modules
module com.myapp {
    requires java.sql;      // ✅ For JDBC
    requires java.xml;      // ✅ For XML APIs
    requires java.desktop;  // ✅ For AWT/Swing
}
----

*10. Not testing module boundaries*

[source,java]
----
// PITFALL - assuming encapsulation works without testing

// Try to access internal package from outside
import com.myapp.internal.Helper;  // Should fail!

// FIX - write tests that verify module boundaries
@Test
public void testModuleEncapsulation() {
    // Attempt to access non-exported package should fail
    assertThrows(IllegalAccessException.class, () -> {
        Class.forName("com.myapp.internal.Helper");
    });
}
----

== Collection Factory Methods

==== Convenient Immutable Collections

.Overview / What it is
[%collapsible]
====
Java 9 introduced *factory methods* for creating immutable collections in a concise way. Think of them as *shorthand for creating read-only collections* that you can't accidentally modify.

*Core Concept:*

Instead of verbose builders or `Collections.unmodifiableX()` wrappers, use simple factory methods:

[source,java]
----
// Old way - verbose and mutable by default
List<String> list = new ArrayList<>();
list.add("Java");
list.add("Python");
list.add("Kotlin");
List<String> immutable = Collections.unmodifiableList(list);

// New way - concise and immutable
List<String> list = List.of("Java", "Python", "Kotlin");
----

*Factory Methods:*

- *List.of()*: Create immutable lists
- *Set.of()*: Create immutable sets
- *Map.of()*: Create immutable maps (up to 10 entries)
- *Map.ofEntries()*: Create immutable maps (any size)

*Key Characteristics:*

1. *Immutable*: Cannot add, remove, or replace elements
2. *Null-hostile*: Reject null keys/values (throws NullPointerException)
3. *Value-based*: Should not rely on identity
4. *Iteration order*: Unspecified for Set/Map (List maintains order)
5. *Serializable*: If all elements are serializable

*Mental Model:*

Think of these as *snapshots* or *constants*. Once created, they never change. Perfect for configuration, constants, or data that shouldn't be modified.
====

.Why use it
[%collapsible]
====
*1. Conciseness*

Dramatically reduce boilerplate:

[source,java]
----
// Before Java 9
Map<String, Integer> map = new HashMap<>();
map.put("one", 1);
map.put("two", 2);
map.put("three", 3);
Map<String, Integer> immutable = Collections.unmodifiableMap(map);

// Java 9+
Map<String, Integer> map = Map.of(
    "one", 1,
    "two", 2,
    "three", 3
);
----

*2. Safety*

Immutability prevents bugs:

[source,java]
----
// Old way - easy to accidentally modify
List<String> allowed = getDefaultRoles();
allowed.add("ADMIN");  // Oops! Modified shared list

// New way - modification attempts throw exception
List<String> allowed = List.of("USER", "GUEST");
allowed.add("ADMIN");  // ❌ UnsupportedOperationException
----

*3. Performance*

Optimized implementations:

[source,java]
----
// List.of() uses specialized implementations
// - List.of() → ImmutableCollections.ListN (more compact than ArrayList)
// - Set.of() → optimized hash implementation
// - No defensive copying needed (already immutable)
----

*4. Thread-Safety*

Immutable collections are inherently thread-safe:

[source,java]
----
// Can safely share across threads without synchronization
public static final List<String> SUPPORTED_FORMATS =
    List.of("JSON", "XML", "YAML");
----

*5. Better API Contracts*

Clearly express intent:

[source,java]
----
// API clearly signals: "This list won't change"
public List<String> getSupportedLanguages() {
    return List.of("Java", "Kotlin", "Scala");
}
----
====

.When not to use
[%collapsible]
====
*❌ DON'T Use for:*

*1. Collections That Need Modification*

[source,java]
----
// BAD - need to modify later
List<String> items = List.of("A", "B", "C");
items.add("D");  // ❌ UnsupportedOperationException

// GOOD - use mutable collection
List<String> items = new ArrayList<>(List.of("A", "B", "C"));
items.add("D");  // ✅ Works
----

*2. Large Collections Built Incrementally*

[source,java]
----
// BAD - creating new immutable list each iteration
List<User> users = List.of();
for (ResultSet rs : resultSets) {
    User user = parseUser(rs);
    users = concat(users, user);  // ❌ Inefficient!
}

// GOOD - build then make immutable
List<User> users = new ArrayList<>();
for (ResultSet rs : resultSets) {
    users.add(parseUser(rs));
}
List<User> immutableUsers = List.copyOf(users);  // ✅
----

*3. Collections with Null Values*

[source,java]
----
// BAD - factory methods reject nulls
List<String> items = List.of("A", null, "C");  // ❌ NullPointerException

// GOOD - use traditional collections if nulls are needed
List<String> items = Arrays.asList("A", null, "C");  // ✅
----

*4. Very Large Maps*

[source,java]
----
// BAD - Map.of() limited to 10 entries
Map<String, String> config = Map.of(
    "k1", "v1", "k2", "v2", ..., "k10", "v10",
    "k11", "v11"  // ❌ Too many arguments
);

// GOOD - use Map.ofEntries()
Map<String, String> config = Map.ofEntries(
    entry("k1", "v1"),
    entry("k2", "v2"),
    // ... any number of entries
    entry("k100", "v100")
);
----

*General Rule:* Use factory methods for *small, immutable collections* that represent constants or configuration. Use traditional collections for large, mutable, or null-containing data structures.
====

.Best practices
[%collapsible]
====
*✅ DO:*

- *Use for constants and configuration*
[source,java]
----
public static final Set<String> VALID_STATUSES =
    Set.of("PENDING", "APPROVED", "REJECTED");

public static final Map<String, String> ERROR_MESSAGES = Map.of(
    "404", "Not Found",
    "500", "Internal Server Error",
    "403", "Forbidden"
);
----

- *Use List.copyOf() for defensive copying*
[source,java]
----
private final List<String> items;

public MyClass(List<String> items) {
    this.items = List.copyOf(items);  // ✅ Immutable copy
}
----

- *Use Map.ofEntries() for larger maps*
[source,java]
----
import static java.util.Map.entry;

Map<String, Integer> map = Map.ofEntries(
    entry("one", 1),
    entry("two", 2),
    entry("three", 3),
    // ... more entries
    entry("hundred", 100)
);
----

- *Combine with var for cleaner code* (Java 10+)
[source,java]
----
var supportedFormats = List.of("JSON", "XML", "YAML");
var httpMethods = Set.of("GET", "POST", "PUT", "DELETE");
var statusCodes = Map.of(200, "OK", 404, "Not Found");
----

- *Return from methods to signal immutability*
[source,java]
----
public List<String> getReadOnlyData() {
    return List.of("A", "B", "C");  // ✅ Caller can't modify
}
----

*❌ DON'T:*

- *Don't use if you need null elements*
[source,java]
----
// BAD
List<String> items = List.of("A", null);  // ❌ NullPointerException

// GOOD - check for nulls or use different approach
List<String> items = Stream.of("A", null)
    .filter(Objects::nonNull)
    .collect(Collectors.toUnmodifiableList());
----

- *Don't use for large collections built incrementally*
[source,java]
----
// BAD
List<Integer> numbers = List.of();
for (int i = 0; i < 1000; i++) {
    numbers = concat(numbers, i);  // ❌ Creates 1000 objects!
}

// GOOD
List<Integer> numbers = IntStream.range(0, 1000)
    .boxed()
    .collect(Collectors.toUnmodifiableList());
----

- *Don't try to modify returned collections*
[source,java]
----
List<String> list = List.of("A", "B", "C");
list.add("D");      // ❌ UnsupportedOperationException
list.remove(0);     // ❌ UnsupportedOperationException
list.set(0, "X");   // ❌ UnsupportedOperationException
----

- *Don't assume iteration order for Set/Map*
[source,java]
----
// BAD - assuming order
Set<String> set = Set.of("A", "B", "C");
String first = set.iterator().next();  // ❌ Order unspecified

// GOOD - use List if order matters
List<String> list = List.of("A", "B", "C");
String first = list.get(0);  // ✅ Guaranteed order
----

== Stream API Improvements

==== New Stream Operations

.Overview / What it is
[%collapsible]
====
Java 9 added four new methods to the Stream API that solve common problems more elegantly. Think of them as *power tools* that handle specific patterns you'd otherwise implement manually.

*New Methods:*

1. *takeWhile()*: Take elements while condition is true (then stop)
2. *dropWhile()*: Skip elements while condition is true (then take rest)
3. *ofNullable()*: Create stream from potentially null value
4. *iterate()* (overload): Iterate with termination condition

*Core Concept:*

These methods bridge gaps in the original Stream API, particularly for working with ordered streams and handling edge cases like nulls.

[source,java]
----
// takeWhile() - take until condition fails
Stream.of(2, 4, 6, 7, 8, 10)
    .takeWhile(n -> n % 2 == 0)  // Takes: 2, 4, 6 (stops at 7)
    .forEach(System.out::println);

// dropWhile() - skip until condition fails
Stream.of(2, 4, 6, 7, 8, 10)
    .dropWhile(n -> n % 2 == 0)  // Skips: 2, 4, 6 (keeps: 7, 8, 10)
    .forEach(System.out::println);

// ofNullable() - handle potential null
Stream.ofNullable(getValue())  // Empty stream if null
    .forEach(System.out::println);

// iterate() with condition - like for loop
Stream.iterate(0, n -> n < 10, n -> n + 1)  // 0, 1, 2, ..., 9
    .forEach(System.out::println);
----

*Mental Model:*

- *takeWhile()*: "Give me elements FROM START until predicate fails"
- *dropWhile()*: "Skip elements FROM START until predicate fails, then give me rest"
- *ofNullable()*: "Stream of 0 or 1 element, safely handling null"
- *iterate(condition)*: "for loop as a stream"
====

.Why use it
[%collapsible]
====
*1. takeWhile() - Process Ordered Data Until Condition*

Perfect for sorted streams or reading until delimiter:

[source,java]
----
// Read lines from file until empty line
Files.lines(path)
    .takeWhile(line -> !line.isEmpty())
    .forEach(System.out::println);

// Process sorted list until threshold
List<Integer> sortedScores = List.of(95, 90, 85, 78, 65, 50);
List<Integer> passing = sortedScores.stream()
    .takeWhile(score -> score >= 70)  // Takes: 95, 90, 85, 78
    .collect(Collectors.toList());
----

*2. dropWhile() - Skip Header/Prefix*

Skip unwanted prefix elements:

[source,java]
----
// Skip comment lines at start of file
Files.lines(path)
    .dropWhile(line -> line.startsWith("#"))
    .forEach(System.out::println);

// Skip low values in sorted data
List<Integer> sorted = List.of(1, 2, 5, 10, 20, 50);
List<Integer> significant = sorted.stream()
    .dropWhile(n -> n < 10)  // Skips: 1, 2, 5 (keeps: 10, 20, 50)
    .collect(Collectors.toList());
----

*3. ofNullable() - Null-Safe Stream Operations*

Avoid NullPointerException in stream pipelines:

[source,java]
----
// Before Java 9 - verbose null checking
Optional.ofNullable(user)
    .map(User::getEmail)
    .ifPresent(email -> sendEmail(email));

// Java 9+ - cleaner with Stream.ofNullable()
Stream.ofNullable(user)
    .map(User::getEmail)
    .forEach(this::sendEmail);

// Flatmap with potential nulls
users.stream()
    .flatMap(user -> Stream.ofNullable(user.getAddress()))
    .forEach(System.out::println);
----

*4. iterate(condition) - Bounded Iteration*

Create finite streams without explicit limit():

[source,java]
----
// Before Java 9 - need limit()
Stream.iterate(1, n -> n * 2)
    .limit(10)  // Must remember to limit!
    .forEach(System.out::println);

// Java 9+ - condition built-in
Stream.iterate(1, n -> n <= 1024, n -> n * 2)  // 1, 2, 4, 8, ..., 1024
    .forEach(System.out::println);

// Like a for loop: for(int i=0; i<10; i++)
Stream.iterate(0, i -> i < 10, i -> i + 1)
    .forEach(System.out::println);
----

.When not to use
[%collapsible]
====
*❌ DON'T Use for:*

*1. takeWhile()/dropWhile() on Unordered Streams*

[source,java]
----
// BAD - behavior undefined for unordered streams
Set<Integer> set = Set.of(1, 2, 3, 4, 5);
set.stream()
    .takeWhile(n -> n < 3)  // ❌ Set is unordered - unpredictable!
    .forEach(System.out::println);

// GOOD - use filter() instead
set.stream()
    .filter(n -> n < 3)  // ✅ Works on unordered streams
    .forEach(System.out::println);
----

*2. takeWhile() When You Need All Matching Elements*

[source,java]
----
// BAD - takeWhile() stops at first non-match
List<Integer> numbers = List.of(2, 4, 5, 6, 8);
numbers.stream()
    .takeWhile(n -> n % 2 == 0)  // Only gets: 2, 4 (stops at 5)
    .forEach(System.out::println);

// GOOD - use filter() to get all matches
numbers.stream()
    .filter(n -> n % 2 == 0)  // Gets: 2, 4, 6, 8
    .forEach(System.out::println);
----

*3. ofNullable() for Collections*

[source,java]
----
// BAD - ofNullable() for collection
List<String> maybeNull = getList();  // Might return null
Stream.ofNullable(maybeNull)  // ❌ Stream of one List or empty
    .forEach(System.out::println);

// GOOD - use orElse with stream()
Optional.ofNullable(maybeNull)
    .orElse(Collections.emptyList())
    .stream()  // ✅ Stream of elements
    .forEach(System.out::println);
----

*General Rule:* Use these methods for their *specific use cases*. Don't force them where traditional Stream operations (`filter()`, `limit()`, etc.) are more appropriate.

.Common pitfalls
[%collapsible]
====
*1. Using takeWhile() on unordered streams*

[source,java]
----
// PITFALL - unpredictable behavior on unordered data

Set<Integer> set = new HashSet<>(List.of(1, 2, 3, 4, 5));
List<Integer> result = set.stream()
    .takeWhile(n -> n < 3)  // ❌ Set iteration order is undefined!
    .collect(Collectors.toList());

// FIX - use filter() for unordered streams
List<Integer> result = set.stream()
    .filter(n -> n < 3)  // ✅ Gets all matching elements
    .collect(Collectors.toList());

// Or sort first if you need takeWhile()
List<Integer> result = set.stream()
    .sorted()
    .takeWhile(n -> n < 3)  // ✅ Now ordered
    .collect(Collectors.toList());

*2. Confusing takeWhile() with filter()*

[source,java]
----
// PITFALL - takeWhile() stops at first non-match

List<Integer> numbers = List.of(2, 4, 6, 5, 8, 10);

List<Integer> result = numbers.stream()
    .takeWhile(n -> n % 2 == 0)  // ❌ Gets only: 2, 4, 6 (stops at 5!)
    .collect(Collectors.toList());

// Expected behavior
List<Integer> result = numbers.stream()
    .filter(n -> n % 2 == 0)  // ✅ Gets: 2, 4, 6, 8, 10
    .collect(Collectors.toList());

// When to use takeWhile(): Sorted data with threshold
List<Integer> sorted = List.of(10, 8, 6, 5, 4, 2);
List<Integer> aboveThreshold = sorted.stream()
    .takeWhile(n -> n > 5)  // ✅ Correct: 10, 8, 6
    .collect(Collectors.toList());

*3. Forgetting iterate() condition leads to infinite stream*

[source,java]
----
// PITFALL - infinite stream without termination

// Old iterate() - infinite by default
Stream.iterate(1, n -> n + 1)
    .forEach(System.out::println);  // ❌ Never stops!

// FIX - use new overload with condition
Stream.iterate(1, n -> n <= 100, n -> n + 1)
    .forEach(System.out::println);  // ✅ Stops at 100

// Or use limit()
Stream.iterate(1, n -> n + 1)
    .limit(100)
    .forEach(System.out::println);  // ✅ Also works
----

*4. Using ofNullable() incorrectly for collections*

[source,java]
----
// PITFALL - ofNullable() creates stream of the collection, not elements

List<String> maybeNull = getList();  // Might be null

Stream.ofNullable(maybeNull)
    .forEach(System.out::println);  // ❌ Prints the List object!

// FIX - handle null and stream elements
Optional.ofNullable(maybeNull)
    .orElse(Collections.emptyList())
    .stream()
    .forEach(System.out::println);  // ✅ Prints elements

// Or use orElseGet
Stream.ofNullable(maybeNull)
    .flatMap(Collection::stream)
    .forEach(System.out::println);  // ✅ Also works
----

*5. dropWhile() doesn't evaluate all elements*

[source,java]
----
// PITFALL - dropWhile() only checks until condition fails

List<String> items = List.of("# comment", "# note", "data", "# another");

List<String> result = items.stream()
    .dropWhile(s -> s.startsWith("#"))
    .collect(Collectors.toList());
// Result: ["data", "# another"]  - "# another" is included!

// FIX - use filter() to remove ALL comments
List<String> result = items.stream()
    .filter(s -> !s.startsWith("#"))
    .collect(Collectors.toList());
// Result: ["data"]  ✅
----

*6. Parallel streams with takeWhile()/dropWhile()*

[source,java]
----
// PITFALL - unpredictable with parallel streams

List<Integer> numbers = IntStream.range(1, 100).boxed().collect(Collectors.toList());

List<Integer> result = numbers.parallelStream()  // ❌ Parallel!
    .takeWhile(n -> n < 50)
    .collect(Collectors.toList());
// Behavior is non-deterministic with parallel streams

// FIX - use sequential stream
List<Integer> result = numbers.stream()  // ✅ Sequential
    .takeWhile(n -> n < 50)
    .collect(Collectors.toList());
----

*7. Not realizing takeWhile() is short-circuiting*

[source,java]
----
// PITFALL - expensive operation after takeWhile() still executes

numbers.stream()
    .peek(n -> System.out.println("Processing: " + n))
    .takeWhile(n -> n < 5)  // Stops at 5
    .map(this::expensiveOperation)  // ❌ Still calls for all before takeWhile()!
    .forEach(System.out::println);

// FIX - put expensive operations after takeWhile()
numbers.stream()
    .takeWhile(n -> n < 5)  // Filter first
    .peek(n -> System.out.println("Processing: " + n))
    .map(this::expensiveOperation)  // ✅ Only processes filtered elements
    .forEach(System.out::println);
----

*8. Misunderstanding iterate() parameter order*

[source,java]
----
// PITFALL - wrong parameter order

// Correct order: seed, hasNext, next
Stream.iterate(0, n -> n < 10, n -> n + 1)  // ✅

// Common mistake: seed, next, hasNext
// Stream.iterate(0, n -> n + 1, n -> n < 10)  // ❌ Won't compile (wrong types)
----

*9. Using ofNullable() in wrong context*

[source,java]
----
// PITFALL - ofNullable() when you need empty stream

// Creating stream that might be null
Stream<String> stream = condition ? Stream.of("A", "B") : null;

Stream.ofNullable(stream)  // ❌ Stream of one Stream or empty
    .forEach(System.out::println);

// FIX - use Optional or conditional
Stream<String> stream = condition ? Stream.of("A", "B") : Stream.empty();
stream.forEach(System.out::println);  // ✅
----

*10. Not considering performance of takeWhile()/dropWhile()*

[source,java]
----
// PITFALL - using takeWhile() on unsorted data

// Unsorted list - takeWhile() doesn't give you all matches
List<Integer> unsorted = List.of(5, 2, 8, 1, 9, 3);

// Inefficient - sorts entire list
List<Integer> result = unsorted.stream()
    .sorted()  // O(n log n)
    .takeWhile(n -> n < 5)
    .collect(Collectors.toList());

// More efficient - just filter
List<Integer> result = unsorted.stream()
    .filter(n -> n < 5)  // O(n)
    .collect(Collectors.toList());
----

== Optional Improvements

==== New Optional Methods

.Overview / What it is
[%collapsible]
====
Java 9 enhanced the `Optional` class with three powerful new methods that make working with optional values even more elegant. Think of these as *missing puzzle pieces* that complete common Optional usage patterns.

*New Methods:*

1. *ifPresentOrElse()*: Execute one action if present, another if empty
2. *or()*: Provide alternative Optional if empty
3. *stream()*: Convert Optional to Stream (0 or 1 element)

*Core Concept:*

These methods fill gaps in the original Optional API, particularly for handling the "else" case and chaining Optional operations more fluently.

[source,java]
----
// ifPresentOrElse() - handle both cases
optional.ifPresentOrElse(
    value -> System.out.println("Found: " + value),
    () -> System.out.println("Not found")
);

// or() - fallback to another Optional
Optional<String> result = primaryOptional
    .or(() -> secondaryOptional)
    .or(() -> tertiaryOptional);

// stream() - use Optional in Stream pipelines
users.stream()
    .map(User::getEmail)  // Returns Optional<String>
    .flatMap(Optional::stream)  // Flatten to stream of values
    .forEach(System.out::println);
----

*Mental Model:*

- *ifPresentOrElse()*: "Do this if present, otherwise do that"
- *or()*: "Try this Optional, if empty try the next one" (lazy Optional chaining)
- *stream()*: "Treat Optional as a 0-or-1 element stream"
====

.Why use it
[%collapsible]
====
*1. ifPresentOrElse() - Handle Both Branches Elegantly*

Before Java 9, handling both present and absent cases was awkward:

[source,java]
----
// Before Java 9 - awkward
if (optional.isPresent()) {
    System.out.println("Value: " + optional.get());
} else {
    System.out.println("No value");
}

// Java 9+ - fluent
optional.ifPresentOrElse(
    value -> System.out.println("Value: " + value),
    () -> System.out.println("No value")
);
----

Perfect for side effects with both cases:

[source,java]
----
userOptional.ifPresentOrElse(
    user -> sendWelcomeEmail(user),
    () -> logMissingUser()
);
----

*2. or() - Lazy Optional Fallback Chain*

Chain multiple Optional sources without eager evaluation:

[source,java]
----
// Before Java 9 - eager evaluation or verbose
Optional<User> user = findInCache(id);
if (user.isEmpty()) {
    user = findInDatabase(id);  // Always executes!
}
if (user.isEmpty()) {
    user = findInBackupDatabase(id);
}

// Java 9+ - lazy evaluation
Optional<User> user = findInCache(id)
    .or(() -> findInDatabase(id))  // Only if cache is empty
    .or(() -> findInBackupDatabase(id));  // Only if both above are empty
----

*3. stream() - Integrate Optional with Stream API*

Flatten streams of Optional values elegantly:

[source,java]
----
// Before Java 9 - verbose filtering
List<String> emails = users.stream()
    .map(User::getEmail)  // Stream<Optional<String>>
    .filter(Optional::isPresent)
    .map(Optional::get)
    .collect(Collectors.toList());

// Java 9+ - elegant flatMap
List<String> emails = users.stream()
    .map(User::getEmail)
    .flatMap(Optional::stream)  // Flattens Optional to Stream
    .collect(Collectors.toList());
----

*4. Combine All Three for Powerful Patterns*

[source,java]
----
findUser(id)
    .or(() -> findInBackup(id))  // Try backup
    .stream()  // Convert to stream
    .peek(user -> System.out.println("Found: " + user))
    .findFirst()
    .ifPresentOrElse(
        user -> processUser(user),
        () -> createDefaultUser()
    );
----
====

.When not to use
[%collapsible]
====
*❌ DON'T Use for:*

*1. ifPresentOrElse() When You Don't Need Both Actions*

[source,java]
----
// BAD - using ifPresentOrElse() with empty action
optional.ifPresentOrElse(
    value -> process(value),
    () -> { /* do nothing */ }  // ❌ Unnecessary
);

// GOOD - use ifPresent()
optional.ifPresent(value -> process(value));  // ✅
----

*2. or() When You Have a Value (Not Optional)*

[source,java]
----
// BAD - wrapping value in Optional just for or()
optional.or(() -> Optional.of(defaultValue))  // ❌ Verbose
    .get();

// GOOD - use orElse()
optional.orElse(defaultValue);  // ✅
----

*3. stream() for Simple Present Check*

[source,java]
----
// BAD - overcomplicated
boolean hasValue = optional.stream()
    .findAny()
    .isPresent();  // ❌ Too complex

// GOOD - use isPresent()
boolean hasValue = optional.isPresent();  // ✅
----

*4. Chaining or() When orElse() Suffices*

[source,java]
----
// BAD - or() for static default
optional.or(() -> Optional.of("default"));  // ❌ Unnecessary Optional

// GOOD - use orElse()
optional.orElse("default");  // ✅
----

*General Rule:* Use these new methods when they make the code *more readable* and *more functional*. Don't force them where simpler Optional methods work better.
====

.Best practices
[%collapsible]
====
*✅ DO:*

- *Use ifPresentOrElse() for side effects with both branches*
[source,java]
----
userRepository.findById(id)
    .ifPresentOrElse(
        user -> auditLog.log("User found: " + user.getId()),
        () -> auditLog.log("User not found: " + id)
    );
----

- *Use or() for lazy Optional chaining*
[source,java]
----
// Multiple data sources with lazy evaluation
Optional<Config> config = loadFromFile()
    .or(() -> loadFromEnvironment())
    .or(() -> loadFromDefaults());
----

- *Use stream() to flatten Optional in Stream pipelines*
[source,java]
----
List<Address> addresses = users.stream()
    .map(User::getAddress)  // Stream<Optional<Address>>
    .flatMap(Optional::stream)  // Stream<Address>
    .collect(Collectors.toList());
----

- *Combine or() with map() for transformation chains*
[source,java]
----
String result = getPrimaryValue()
    .map(String::toUpperCase)
    .or(() -> getSecondaryValue().map(String::toLowerCase))
    .orElse("DEFAULT");
----

- *Use stream() with filter and findFirst*
[source,java]
----
Optional<User> admin = users.stream()
    .map(this::validateUser)  // Stream<Optional<User>>
    .flatMap(Optional::stream)
    .filter(User::isAdmin)
    .findFirst();
----

*❌ DON'T:*

- *Don't use ifPresentOrElse() for value transformation*
[source,java]
----
// BAD - side effects for transformation
String result;
optional.ifPresentOrElse(
    value -> result = value.toUpperCase(),  // ❌ Mutating external variable
    () -> result = "DEFAULT"
);

// GOOD - use map/orElse for transformation
String result = optional
    .map(String::toUpperCase)
    .orElse("DEFAULT");  // ✅
----

- *Don't use or() when orElseGet() is clearer*
[source,java]
----
// BAD - wrapping in Optional
String value = optional
    .or(() -> Optional.of(computeDefault()))
    .get();  // ❌

// GOOD - use orElseGet()
String value = optional
    .orElseGet(() -> computeDefault());  // ✅
----

- *Don't chain or() excessively*
[source,java]
----
// BAD - too many fallbacks
optional
    .or(() -> opt1)
    .or(() -> opt2)
    .or(() -> opt3)
    .or(() -> opt4)
    .or(() -> opt5);  // ❌ Hard to read

// GOOD - consider a different approach
Stream.of(optional, opt1, opt2, opt3, opt4, opt5)
    .flatMap(Optional::stream)
    .findFirst();  // ✅
----

- *Don't use stream() when you don't need Stream operations*
[source,java]
----
// BAD - unnecessary stream
optional.stream()
    .forEach(System.out::println);  // ❌

// GOOD - use ifPresent()
optional.ifPresent(System.out::println);  // ✅
----

== Private Interface Methods

==== Code Reuse in Interfaces

.Overview / What it is
[%collapsible]
====
Java 9 allows interfaces to have *private methods* (both instance and static). Think of them as *helper methods that reduce duplication* in default and static interface methods without exposing implementation details.

*Core Concept:*

Before Java 9, you couldn't extract common code from default/static methods in interfaces. Now you can create private helper methods for code reuse.

[source,java]
----
public interface Calculator {

    // Public default methods
    default int addPositive(int a, int b) {
        validate(a);
        validate(b);
        return a + b;
    }

    default int multiplyPositive(int a, int b) {
        validate(a);
        validate(b);
        return a * b;
    }

    // Private helper method - reduces duplication
    private void validate(int number) {
        if (number < 0) {
            throw new IllegalArgumentException("Only positive numbers allowed");
        }
    }
}
----

*What You Can Do:*

1. *Private instance methods*: Used by default methods
2. *Private static methods*: Used by static and default methods

*What You CANNOT Do:*

- Private methods cannot be abstract (they must have a body)
- Private methods are not inherited or visible to implementing classes
- Private methods cannot be called from outside the interface

*Mental Model:*

Think of private interface methods as *internal implementation details* that help keep default/static methods DRY (Don't Repeat Yourself), just like private methods in classes.

[%collapsible]
====
*1. Eliminate Code Duplication*

Extract common logic from default methods:

        validateUser(user);
        // ... create user

        // ... update user

Private methods don't leak to implementing classes:

        List<String> cleaned = cleanData(rawData);
        List<String> validated = validateData(cleaned);

*Example 5: Configuration Builder*

public interface ConfigBuilder {

    default Config buildConfig(Map<String, String> properties) {
        validateProperties(properties);
        Map<String, String> normalized = normalizeKeys(properties);
        Map<String, String> withDefaults = applyDefaults(normalized);
        return new Config(withDefaults);
    }

    private void validateProperties(Map<String, String> properties) {
        if (properties == null || properties.isEmpty()) {
            throw new IllegalArgumentException("Properties cannot be null or empty");
        }

        validateRequiredKeys(properties);
    }

    private void validateRequiredKeys(Map<String, String> properties) {
        List<String> required = List.of("app.name", "app.version");
        for (String key : required) {
            if (!properties.containsKey(key)) {
                throw new IllegalArgumentException("Missing required property: " + key);
            }
        }
    }

    private Map<String, String> normalizeKeys(Map<String, String> properties) {
        return properties.entrySet().stream()
            .collect(Collectors.toMap(
                e -> e.getKey().toLowerCase(),
                Map.Entry::getValue
            ));
    }

    private Map<String, String> applyDefaults(Map<String, String> properties) {
        Map<String, String> result = new HashMap<>(properties);
        result.putIfAbsent("app.env", "development");
        result.putIfAbsent("app.port", "8080");
        return result;
    }
}

class Config {
    private final Map<String, String> properties;
    Config(Map<String, String> properties) {
        this.properties = properties;
    }
    public String get(String key) { return properties.get(key); }
}

== Try-with-resources Improvements

==== Effectively Final Resources

.Overview / What it is
[%collapsible]
====
Java 9 allows you to use *effectively final* variables in try-with-resources statements without re-declaring them. Think of it as *removing unnecessary boilerplate* when you already have a resource variable.

*Core Concept:*

Before Java 9, you had to declare resources inside the try-with-resources parentheses. Now, if you have a final or effectively final variable, you can reference it directly.

[source,java]
----
// Before Java 9 - must declare inside try()
BufferedReader reader1 = new BufferedReader(new FileReader("file.txt"));
try (BufferedReader reader2 = reader1) {  // ❌ Redundant variable
    return reader2.readLine();
}

// Java 9+ - use existing effectively final variable
BufferedReader reader = new BufferedReader(new FileReader("file.txt"));
try (reader) {  // ✅ Direct reference
    return reader.readLine();
}
----

*Requirements:*

- Variable must be *final* or *effectively final* (not reassigned after initialization)
- Variable must be initialized before the try-with-resources statement
- Variable's type must implement `AutoCloseable` or `Closeable`

*Mental Model:*

Think of this as "use what you already have" - if you have a resource variable that won't change, just use it directly in try-with-resources without creating a redundant copy.
====

.Why use it
[%collapsible]
====
*1. Reduced Boilerplate*

Eliminates redundant variable declarations:

[source,java]
----
// Before Java 9
Scanner scanner = new Scanner(new File("data.txt"));
try (Scanner s = scanner) {  // ❌ Extra variable
    while (s.hasNextLine()) {
        System.out.println(s.nextLine());
    }
}

// Java 9+
Scanner scanner = new Scanner(new File("data.txt"));
try (scanner) {  // ✅ Cleaner
    while (scanner.hasNextLine()) {
        System.out.println(scanner.nextLine());
    }
}

Use the same variable name throughout:

[source,java]

// Before Java 9 - confusing naming
Connection connection = getConnection();

// Java 9+ - consistent naming
Connection connection = getConnection();
try (connection) {  // ✅ Same name throughout

.When not to use
[%collapsible]
====
*❌ DON'T Use for:*

*1. Variables That Are Not Effectively Final*

[source,java]
----

}

// GOOD - declare in try() if you need to reassign
try (BufferedReader reader = new BufferedReader(new FileReader("file.txt"))) {
    return reader.readLine();
}
----

*2. When Resource Creation Can Fail*

[source,java]
----
// BAD - if creation fails, nothing to close
FileInputStream input = new FileInputStream("file.txt");  // Can throw!
FileOutputStream output = new FileOutputStream("out.txt");  // If this throws, input not closed!
try (input; output) {
    // ...
}

// GOOD - declare inside try() for exception safety
try (FileInputStream input = new FileInputStream("file.txt");
     FileOutputStream output = new FileOutputStream("out.txt")) {
    // If second fails, first is still closed ✅
}
----

*3. When Variable Might Be Null*

[source,java]
----
// BAD - potential NullPointerException
BufferedReader reader = condition ? new BufferedReader(...) : null;
try (reader) {  // ❌ NPE if reader is null
    // ...
}

// GOOD - check for null first
BufferedReader reader = condition ? new BufferedReader(...) : null;
if (reader != null) {
    try (reader) {
        // ...
    }
}
----

*4. When You Need Different Scope*

[source,java]
----
// BAD - resource needed after try block
BufferedReader reader = new BufferedReader(new FileReader("file.txt"));
try (reader) {
    // Process
}
// Can't use reader here - it's closed! ❌

// GOOD - don't use try-with-resources if you need it open
BufferedReader reader = new BufferedReader(new FileReader("file.txt"));
try {
    // Process
} finally {
    if (reader != null) reader.close();
}
// Can still use reader if you need to
----

*General Rule:* Use this feature when you have a *truly effectively final* resource variable that you want to auto-close. For complex scenarios with multiple failure points or conditional resources, stick with traditional try-with-resources declarations.
====

.Best practices
[%collapsible]
====
*✅ DO:*

- *Use for method parameters*
[source,java]
----
public String readFirstLine(BufferedReader reader) throws IOException {
    try (reader) {  // ✅ Parameter is effectively final
        return reader.readLine();
    }
}
----

- *Use for final variables*
[source,java]
----
final Connection connection = dataSource.getConnection();
try (connection) {  // ✅ Explicit final
    // Use connection
}
----

- *Combine multiple existing resources*
[source,java]
----
Scanner scanner = new Scanner(System.in);
Formatter formatter = new Formatter();

try (scanner; formatter) {  // ✅ Both closed
    String input = scanner.nextLine();
    formatter.format("Input: %s%n", input);
}
----

- *Use with returned resources*
[source,java]
----
Connection conn = connectionPool.getConnection();
try (conn) {  // ✅ Pool connection auto-returned
    // Use connection
}
----

*❌ DON'T:*

- *Don't use if variable might be reassigned*
[source,java]
----
// BAD
Connection conn = getConnection();
if (needsSpecialConnection()) {
    conn = getSpecialConnection();  // ❌ Reassigned!
}
try (conn) {  // Compilation error
    // ...
}

// GOOD - declare inside try
Connection conn;
if (needsSpecialConnection()) {
    conn = getSpecialConnection();
} else {
    conn = getConnection();
}
try (conn) {  // ✅ Works if no reassignment after this
    // ...
}
----

- *Don't use if resource creation might fail in sequence*
[source,java]
----
// BAD - exception safety issue
InputStream in = new FileInputStream("in.txt");   // If next line fails...
OutputStream out = new FileOutputStream("out.txt"); // ...in not closed!
try (in; out) {
    // ...
}

// GOOD - all-or-nothing creation
try (InputStream in = new FileInputStream("in.txt");
     OutputStream out = new FileOutputStream("out.txt")) {
    // ✅ If second fails, first still closed
}
----

- *Don't mix declaration styles unnecessarily*
[source,java]
----
// CONFUSING - inconsistent style
Connection conn = getConnection();
try (conn; PreparedStatement stmt = conn.prepareStatement("...")) {
    // ❌ Mixing styles is confusing
}

// BETTER - consistent style
try (Connection conn = getConnection();
     PreparedStatement stmt = conn.prepareStatement("...")) {
    // ✅ All declared in try()
}

// ALSO GOOD - all effectively final
Connection conn = getConnection();
PreparedStatement stmt = conn.prepareStatement("...");
try (conn; stmt) {
    // ✅ Consistent: both pre-existing
}
----

- *Don't forget the resource must be AutoCloseable*
[source,java]
----
String notCloseable = "text";
try (notCloseable) {  // ❌ Compilation error: String not AutoCloseable
    // ...
}
----

public void processStream(InputStream input, OutputStream output) throws IOException {

*Example 8: Complex Real-World Scenario*

[source,java]
----
public class DatabaseBackup {

    public void backup(Connection sourceConn, Connection backupConn) throws SQLException {
        String selectQuery = "SELECT * FROM important_data";
        String insertQuery = "INSERT INTO important_data VALUES (?, ?, ?)";

        // Both connections are parameters (effectively final)
        try (sourceConn; backupConn;
             PreparedStatement selectStmt = sourceConn.prepareStatement(selectQuery);
             PreparedStatement insertStmt = backupConn.prepareStatement(insertQuery);
             ResultSet rs = selectStmt.executeQuery()) {

            while (rs.next()) {
                insertStmt.setLong(1, rs.getLong("id"));
                insertStmt.setString(2, rs.getString("name"));
                insertStmt.setString(3, rs.getString("data"));
                insertStmt.executeUpdate();
            }

            backupConn.commit();
            System.out.println("Backup completed successfully");
        }
        // All resources automatically closed in reverse order
    }
}

.Common pitfalls
[%collapsible]
====
*1. Using non-effectively-final variables*

[source,java]
----
// PITFALL - variable reassigned

BufferedReader reader = new BufferedReader(new FileReader("file1.txt"));
reader = new BufferedReader(new FileReader("file2.txt"));  // Reassigned!

try (reader) {  // ❌ Compilation error: variable is not effectively final
    return reader.readLine();
}

// FIX - make it final or don't reassign
final BufferedReader reader = new BufferedReader(new FileReader("file.txt"));
try (reader) {  // ✅ Works
    return reader.readLine();
}
----

*2. Resource creation exception leaves previous resource open*

[source,java]
----
// PITFALL - if second resource fails, first leaks

InputStream input = new FileInputStream("in.txt");    // Opens successfully
OutputStream output = new FileOutputStream("out.txt"); // Throws IOException!
// input is never closed! ❌

try (input; output) {
    // Never reached
}

// FIX - create all resources inside try()
try (InputStream input = new FileInputStream("in.txt");
     OutputStream output = new FileOutputStream("out.txt")) {
    // ✅ If output fails, input still gets closed
}
----

*3. Using null resources*

[source,java]
----
// PITFALL - null resource

BufferedReader reader = condition ? new BufferedReader(...) : null;

try (reader) {  // ❌ NullPointerException if reader is null
    return reader.readLine();
}

// FIX - check for null
BufferedReader reader = condition ? new BufferedReader(...) : null;
if (reader != null) {
    try (reader) {
        return reader.readLine();
    }
}
----

*4. Forgetting variable must be AutoCloseable*

[source,java]
----
// PITFALL - not AutoCloseable

class MyResource {  // Doesn't implement AutoCloseable
    public void doSomething() { }
}

MyResource resource = new MyResource();
try (resource) {  // ❌ Compilation error
    resource.doSomething();
}

// FIX - implement AutoCloseable
class MyResource implements AutoCloseable {
    public void doSomething() { }

    @Override
    public void close() {
        // Cleanup logic
    }
}
----

*5. Closing resources in wrong order*

[source,java]
----
// PITFALL - order matters for dependent resources

InputStream input = new FileInputStream("file.txt");
BufferedInputStream buffered = new BufferedInputStream(input);

try (buffered; input) {  // ❌ Wrong order!
    // When exiting: input closes first, then buffered tries to use closed input
}

// FIX - close in reverse dependency order (or let try-with-resources handle it)
try (input; buffered) {  // ✅ buffered closes first, then input
    // Proper order
}

// BEST - declare inside try(), order handled automatically
try (InputStream input = new FileInputStream("file.txt");
     BufferedInputStream buffered = new BufferedInputStream(input)) {
    // ✅ Closes in correct reverse order automatically
}
----

*6. Modifying variable after try-with-resources*

[source,java]
----
// PITFALL - treating closed resource as still open

BufferedReader reader = new BufferedReader(new FileReader("file.txt"));

try (reader) {
    String firstLine = reader.readLine();
    System.out.println(firstLine);
}
// reader is now closed!

String secondLine = reader.readLine();  // ❌ IOException: Stream closed

// FIX - don't use resource after try block
BufferedReader reader = new BufferedReader(new FileReader("file.txt"));
String firstLine, secondLine;

try (reader) {
    firstLine = reader.readLine();
    secondLine = reader.readLine();  // ✅ Read while still open
    System.out.println(firstLine);
    System.out.println(secondLine);
}
----

*7. Mixing old and new syntax inconsistently*

[source,java]
----
// PITFALL - confusing mix

Connection conn = getConnection();
try (conn; PreparedStatement stmt = conn.prepareStatement("...")) {
    // ❌ Why is conn outside but stmt inside?
}

// BETTER - be consistent
// Option 1: All inside
try (Connection conn = getConnection();
     PreparedStatement stmt = conn.prepareStatement("...")) {
    // ✅ Consistent
}

// Option 2: All outside (if both are effectively final)
Connection conn = getConnection();
PreparedStatement stmt = conn.prepareStatement("...");
try (conn; stmt) {
    // ✅ Also consistent
}
----

*8. Not understanding effectively final*

[source,java]
----
// PITFALL - thinking final keyword is required

// This works! (effectively final)
BufferedReader reader = new BufferedReader(new FileReader("file.txt"));
// Not reassigned anywhere
try (reader) {  // ✅ Works without 'final' keyword
    return reader.readLine();
}

// This doesn't work (not effectively final)
BufferedReader reader2 = new BufferedReader(new FileReader("file1.txt"));
if (condition) {
    reader2 = new BufferedReader(new FileReader("file2.txt"));  // Reassignment!
}
try (reader2) {  // ❌ Not effectively final
    return reader2.readLine();
}
----

*9. Trying to close resources manually*

[source,java]
----
// PITFALL - manual close in try-with-resources

BufferedReader reader = new BufferedReader(new FileReader("file.txt"));

try (reader) {
    String line = reader.readLine();
    reader.close();  // ❌ Don't do this! Will be closed again automatically
    return line;
}
// Double close might cause issues

// FIX - let try-with-resources handle it
try (reader) {
    String line = reader.readLine();
    return line;  // ✅ Auto-closed
}
----

*10. Suppressed exceptions confusion*

[source,java]
----
// PITFALL - not understanding suppressed exceptions

BufferedReader reader = new BufferedReader(new FileReader("file.txt"));

try (reader) {
    processLine(reader.readLine());  // Throws ProcessException
}
// If reader.close() also throws, ProcessException is main, close() exception is suppressed

// Access suppressed exceptions
catch (ProcessException e) {
    Throwable[] suppressed = e.getSuppressed();
    for (Throwable t : suppressed) {
        System.err.println("Suppressed: " + t);
    }
}
----
