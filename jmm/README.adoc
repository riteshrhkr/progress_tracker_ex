== Introduction to Java Memory Model (JMM)

=== Definition and Purpose
The Java Memory Model (JMM) is a specification that guarantees how and when changes made by one thread to shared data become visible to other threads. Because Java is designed to be platform-independent, the JMM abstracts away the underlying hardware memory architectures (which vary significantly across different CPUs and OS platforms) and provides a unified model for multithreaded programming.

=== Problems JMM Solves
In a multi-core environment, threads can execute concurrently and variables might be aggressively cached in CPU registers or local caches. The JMM primarily solves three fundamental problems:

1. **Visibility:** When one thread changes the value of a shared variable, when does that change become visible to other threads? Without JMM guarantees, a thread might read a stale value from its CPU cache indefinitely.
2. **Ordering:** Compilers, JVMs, and CPUs often reorder instructions for performance optimization. JMM establishes rules on what reorderings are permitted and defines mechanisms to prevent them when necessary.
3. **Atomicity:** It defines which operations fall entirely into one indivisible step. For instance, reading/writing 32-bit variables (like `int` or `float`) are guaranteed to be atomic, but operations like `i++` are not.

---

== Happens-Before Relationship

The central concept in the JMM is the *happens-before* relationship. If an action `A` happens-before action `B` (`A -> B`), the JMM guarantees that the results of `A` are visible to `B`, and `A` occurs before `B` in terms of execution order.

=== Rules and Examples
Here are the critical happens-before rules:

1. *Program Order Rule:*  -> In the same thread, code runs step by step. Whatever comes first happens before what comes later.

2. *Monitor Lock Rule:*  -> When one thread releases a lock, the next thread that takes the same lock will see all the changes made before the lock was released.

3. *Volatile Variable Rule:*  -> When a thread updates a `volatile` variable, other threads will always see the latest updated value.

4. *Thread Start Rule:*  -> Everything done before calling `Thread.start()` will be visible to the new thread.

5. *Thread Join Rule:*  -> When one thread finishes and another thread waits using `join()`, the waiting thread will see all the work done by the finished thread.

6. *Transitivity:*  -> If A happens before B, and B happens before C, then A also happens before C.

=== Real-World Scenarios
Consider two threads accessing variables `x` and `v`:

[source,java]
----
int x = 0;
volatile boolean v = false;

// Thread A
x = 42;         // (1)
v = true;       // (2)

// Thread B
if (v == true) { // (3)
    print(x);    // (4)
}
----

---

Because `v` is `volatile`, the write to `v` at (2) happens-before the read at (3). Due to the Program Order Rule, (1) happens-before (2), and (3) happens-before (4). By transitivity, if Thread B sees `v == true`, then (1) happens-before (4). Therefore, Thread B is guaranteed to print `42`.

== Java Memory Structure 

The JVM divides memory into several run-time data areas.

    +---------------------------+    +------------------------------+
    |     Thread-Shared         |    |  Thread-Local (Per Thread)   |
    +---------------------------+    +------------------------------+
    | Heap Memory               |    | PC Register                  |
    | Metaspace (Method Area)   |    | JVM Stack                    |
    |                           |    | Native Method Stack          |
    +---------------------------+    +------------------------------+

=== Core Memory Areas

* **Heap Memory:** Shared across all threads. Used for dynamically allocating instances of objects and arrays. It is the primary target for Garbage Collection.
* **Metaspace (Formerly Method Area):** Shared across all threads. Stores class-level data, metadata, statics, and the runtime constant pool. (Since Java 8, it resides in native OS memory).
* **JVM Stack:** Thread-local. Stores frames which hold local variables, operand stacks, and references to objects in the heap.
* **PC (Program Counter) Register:** Thread-local. Stores the address of the current JVM instruction being executed for the thread.
* **Native Method Stack:** Thread-local. Used for native methods written in languages like C/C++ accessed via JNI.

== Variables and References

=== Primitive vs. Reference Types
* **Primitives** (`int`, `double`, `boolean`, etc.) declared as local variables are stored entirely on the JVM Stack (within the thread's frame). Primitive instance variables (class members) are stored on the Heap alongside their object.
* **Reference Types** as local variables hold a *memory address* stored on the JVM Stack, which points to an Object residing on the Heap.
----
    +----------------+        +----------------------+
    |    Stack       |        |        Heap          |
    +----------------+        +----------------------+
    | p (reference)  | -----> | Object: Person       |
    |                |        |-------Data-----------|
    |                |        | name = "Alice"       |
    |                |        | age  = 25            |
    +----------------+        +----------------------+
----


=== Object Memory Layout
In typical HotSpot (OpenJDK) JVM, an object in the heap consists of:

1. **Mark Word:** (8 bytes on 64-bit) Stores hash codes, GC generation age, and synchronization lock states.
2. **Klass Pointer:** (4 bytes using compressed oops, 8 bytes otherwise) Points to the class metadata in the Metaspace.
3. **Instance Data:** The actual fields of the object.
4. **Padding:** Used to align the object's total size to a multiple of 8 bytes for CPU efficiency.

=== Pass-by-Value Concept
Java is strictly **pass-by-value**. When an object is passed to a method, the *value of the reference* (the memory address) is copied. You can modify the object's fields via the reference, but reassigning the reference variable itself to a new object does not affect the caller's reference.

== Garbage Collection and Memory Management

Garbage Collection (GC) automatically removes objects that are no longer reachable, freeing memory and preventing leaks.

=== Reachability

GC starts from **GC Roots**:
- Active local variables in stack
- Static variables
- Active JNI references  

It traverses the object graph. Objects not reachable from any GC Root are considered garbage.

    +-------------------+
    |     GC Roots       |
    |  (Stack, Static)   |
    +---------+---------+
            |
            X
            v
    +-------------------+
    |   Object Graph    |
    |   (Heap Objects)  |
    +---------+---------+
            |
            v
    Unreachable objects → Eligible for GC

=== Generational GC
 
The heap is divided into **Young Generation** and **Old Generation**:

    +-----------------------------+
    |        Young Generation      |
    |  +-------+  +-------------+ |
    |  | Eden  |  | Survivor S0| |
    |  +-------+  +-------------+ |
    |           +-------------+   |
    |           | Survivor S1|   |
    |           +-------------+   |
    +-----------------------------+
    |       Old Generation         |
    +-----------------------------+

- **Young Generation:**  
  - New objects are allocated here (Eden).  
  - Short-lived objects die quickly and are collected in **Minor GC**.  

- **Old Generation (Tenured):**  
  - Objects that survive multiple Minor GCs are promoted here.  
  - Collected less frequently by **Major GC**.
  - As long lived and big objects live here, The space of Old Generation is larger.

=== Types of GC

GC is an expensive operation. When it starts JVM pauses the world means all other operations are paused for that moment. There are three main types of GC:

* **Minor GC:**  
  - Cleans Young Generation only.  
  - Fast, frequent, stops the world briefly.

* **Major GC (Old GC):**  
  - Cleans Old Generation.  
  - Slower, less frequent, longer pauses.

* **Full GC:**  
  - Cleans both Young and Old Generations (sometimes Metaspace too).  
  - Expensive operation; avoid frequent Full GCs.

=== GC Process Diagram (Simplified)

    +-----------------------------+
    |        Eden Space           |
    |   +---------------------+   |
    |   |  New Objects Here   |   |
    |   +---------------------+   |
    +-----------------------------+
            |
            v (Minor GC)
    +-----------------------------+
    |   Survivor Spaces (S0/S1)   |
    |  Objects surviving Minor GC  |
    +-----------------------------+
            |
            v (Promotion)
    +-----------------------------+
    |       Old Generation        |
    | Long-lived objects retained |
    +-----------------------------+

=== Different GC Algorithms
=== Different GC Algorithms

* **Serial GC:** Single-threaded, simple. As it run only on the main thread, It pause for long time.
* **Parallel GC:**  Same as Serial GC but with Multi-threaded.
* **CMS (Concurrent Mark Sweep):** Reduces pause times, mostly concurrent.  
* **G1 (Garbage-First):** Divides heap into regions, balances pause and throughput.  
* **ZGC / Shenandoah:** Low-latency, mostly concurrent, minimal stop-the-world pauses.

NOTE:  From **JDK 17 onwards**, **G1 GC** is the default garbage collector.  
    It is chosen because it **balances throughput and pause times** for most applications.
    Unlike older collectors (Serial, Parallel, CMS), G1 **divides the heap into regions**, allowing **predictable, shorter GC pauses** even with large heaps.

== Types of References

The `java.lang.ref` package defines different reference types to allow interaction with the GC.

* **Strong Reference:** The default (e.g., `Object obj = new Object()`). Prevents the object from being garbage collected.
* **Soft Reference (`SoftReference`):** GC will only clear soft references if the JVM is running low on memory (right before throwing an `OutOfMemoryError`). Highly useful for memory-sensitive caches.
* **Weak Reference (`WeakReference`):** GC reclaims the referent at the very next GC cycle, regardless of memory availability. Often used in `WeakHashMap` to prevent memory leaks when keys are no longer referenced.
* **Phantom Reference (`PhantomReference`):** Enqueued when the object is physically removed from memory. Used for post-mortem cleanups as a safer, more flexible alternative to `Object.finalize()`.

== Concurrency Fundamentals

=== Thread Lifecycle
A Java thread has various states:
* **NEW:** Created but not yet started (`start()` not called).
* **RUNNABLE:** Executing in the JVM (may be waiting for OS resources like CPU).
* **BLOCKED:** Waiting to acquire a monitor lock to enter/re-enter a synchronized block.
* **WAITING:** Waiting indefinitely for another thread to perform a specific action (e.g., via `Object.wait()`).
* **TIMED_WAITING:** Waiting for a specified time (e.g., `Thread.sleep()`).
* **TERMINATED:** Thread has completed execution.

=== Race Conditions and Visibility Issues
* **Race Condition:** Occurs when multiple threads mutate shared state, and the outcome depends on thread scheduling. Operations like `counter++` are not atomic (Read -> Modify -> Write).
* **Visibility Issue:** Thread A updates a variable, but Thread B continues to see the old version because Thread B's CPU core caches the value.

---

== Volatile Keyword

The `volatile` keyword acts as a modifier for fields.

=== Guarantees
1. **Visibility Guarantee:** Any write to a `volatile` variable is immediately flushed to main memory, and any subsequent read bypasses CPU caches to read directly from main memory.
2. **Ordering Guarantee:** It establishes a "happens-before" edge and prevents instruction reordering. E.g., instructions before a volatile write cannot be reordered after it.

=== Limitations
`volatile` **does not guarantee atomicity**. Operations like `count++` are unsafe even if `count` is volatile. Use `volatile` only for flags (e.g., stopping a loop) or in scenarios combined with synchronization (like Double-Checked Locking).

---

== Atomic Classes

For lock-free thread-safe mathematical operations, use the `java.util.concurrent.atomic` package.

=== Compare-And-Swap (CAS)
Atomic classes use CAS—a CPU-level instruction that updates a variable atomically. It works by saying: "Update variable V to new value N, provided the current value is still expected value E. If not, fail and try again."

=== Example
[source,java]
----
import java.util.concurrent.atomic.AtomicInteger;

public class AtomicCounter {
    private AtomicInteger count = new AtomicInteger(0);

    public void increment() {
        count.incrementAndGet(); // Thread-safe, atomic, without locking!
    }
}
----

== Synchronization and Locks

=== `synchronized` Keyword
Using `synchronized` ensures that only one thread can execute a block of code at a time using an **intrinsic monitor lock**.
* Provides both atomicity (mutual exclusion) and visibility (cache flush on exit).
* Drawback: Threads waiting for the lock are BLOCKED and context-switched heavily.

=== ReentrantLock and ReadWriteLock
Located in `java.util.concurrent.locks`, these provide advanced lock management.
* `ReentrantLock`: Similar to `synchronized` but allows fairness policies, interruptible waitings, and try-locking (`tryLock()`).
* `ReadWriteLock`: Allows multiple threads to read concurrently but requires an exclusive lock for writing. Useful in read-heavy scenarios.

== Advanced Topics

* **Instruction Reordering:** Compilers/CPUs reorder instructions strictly within single-thread semantic rules. They cannot anticipate thread interactions, causing bugs unless synchronized.
* **Memory Barriers (Fences):** CPU instructions (StoreStore, StoreLoad, LoadLoad, LoadStore) inserted by the JVM compiler to enforce ordering and prevent CPU caching discrepancies. `volatile` utilizes these heavily.
* **False Sharing:** Two independent variables reside on the same CPU cache line (typically 64 bytes). If Thread A updates var 1 and Thread B updates var 2, they constantly invalidate each other's cache line causing massive performance degradation. Java mitigates this via the `@Contended` annotation to pad variables.
* **Escape Analysis:** A JVM optimization where, if an object allocated in a method does not "escape" the method, it may be allocated directly on the Stack, saving heap space and GC overhead.
* **ThreadLocal:** Provides thread-confined variables. E.g., `ThreadLocal<SimpleDateFormat>` ensures each thread has its own formatter instance, preventing concurrency bugs.
* **Double-Checked Locking (DCL):** A pattern for thread-safe lazy initialization. Requires `volatile` to prevent returning partially initialized objects due to reordering. 
  _See the associated Java file:_ link:DoubleCheckedLocking.java[DoubleCheckedLocking.java]
* **VarHandle (Java 9+):** A safer, strongly-typed alternative to `Unsafe` for dynamic memory manipulations and fine-grained atomic operations.
* **Unsafe:** A low-level class (`sun.misc.Unsafe`) used in high-performance frameworks to manipulate raw memory bypass the JMM bounds. *Not recommended for standard app dev.*

== Hardware-Level Concepts

=== CPU Cache Hierarchy
Accessing main memory (RAM) is extremely slow compared to CPU speed. Therefore, CPUs use caching:
* **L1 Cache:** Immediate to the core, extremely fast, very small.
* **L2 Cache:** Slightly slower, larger, usually per core.
* **L3 Cache:** Shared among all cores.

=== Cache Coherence
Because each core has local caches, having multiple cores modifying the same memory leads to conflict. Protocols like MESI (Modified, Exclusive, Shared, Invalid) ensure that if one core modifies a cache line, other cores are notified to mark their copy as 'Invalid'.

=== Impact on the JMM
The JMM shields the developer from dealing with MESI and manual memory barriers. Writing `synchronized` maps down to acquiring OS mutexes and CPU cache synchronization natively.

== Common Concurrency Problems

1. **Race Conditions:** "Check-then-act" scenarios. E.g., `if(map.containsKey(str)) map.remove(str);` is fundamentally unsafe across threads.
2. **Deadlocks:** Thread A holds Lock 1 waiting for Lock 2. Thread B holds Lock 2 waiting for Lock 1. Both wait forever.
3. **Livelocks:** Threads continually alter their state in response to other threads, doing useless work but never making actual progress.
4. **Memory Consistency Errors:** When threads see inconsistent, stale views of shared data, primarily due to lacking `volatile` or memory boundaries.

---

== Best Practices

1. **Favor Immutability:** Declare fields as `final`. Immutable objects require absolutely ZERO synchronization and are inherently thread-safe.
2. **Use Higher-Level Utilities:** Prefer `java.util.concurrent` (e.g., `ConcurrentHashMap`, `CountDownLatch`, `ExecutorService`) over manual `wait()`/`notify()` and thread management.
3. **Minimize Lock Scope:** When `synchronized` is necessary, synchronize only the critical sections, not entire methods, to avoid bottlenecks.
4. **Choose the Right Tool:**
   * Does it need mutual exclusion? Use Locks / `synchronized`.
   * Is it just a simple counter or state toggle? Use `AtomicInteger` / `AtomicBoolean`.
   * Is it an indicator flag updated by one thread and read by others? Use `volatile`.

== Interview Questions Section

1. **What is the difference between `volatile` and `AtomicInteger`?**
_Answer:_ `volatile` only guarantees visibility and prevents instruction reordering. It doesn't make composite operations like `++` atomic. `AtomicInteger` uses CAS to guarantee both visibility and atomicity for mathematical transformations.

2. **Can you explain the "happens-before" guarantee?**
_Answer:_ It's the core JMM rule that dictates when thread A's memory modifications become safely visible to thread B. If action X happens-before action Y, thread Y is guaranteed to see the results of X.

3. **Why do we need the `volatile` keyword in Double-Checked Locking?**
_Answer:_ Because creating an object involves allocating memory, calling the constructor, and assigning the reference. The compiler might reorder these, assigning the reference before the constructor runs. `volatile` prevents this reordering, ensuring the object is fully constructed before any thread sees a non-null instance.

4. **What is 'False Sharing' and how does Java solve it?**
_Answer:_ False sharing occurs when two independent thread variables share the same CPU cache line, causing constant cache invalidation and slowing down performance heavily. Java solves it via memory padding, such as using the `@Contended` annotation, ensuring variables reside on independent cache lines.

5. **Where are object references stored versus the actual objects?**
_Answer:_ Local reference variables are stored on the thread's Stack memory. The actual object they point to resides in the Heap memory.

=== Summary
The Java Memory Model is the underlying contract ensuring multithreaded predictability in Java. By mastering concepts such as `happens-before`, `volatile`, atomics, and intrinsic locks, developers can write robust, high-performance concurrent applications while evading subtle, difficult-to-reproduce threading bugs.
